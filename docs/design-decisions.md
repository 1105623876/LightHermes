# LightHermes 设计决策

## 1. 优先生成 Skill 而不是 Tool

**问题**: 技能生成的沙箱验证会引入延迟,影响自进化系统的响应速度。

**决策**: 自进化系统优先生成 Markdown 技能(type: skill),而不是 Python 插件(type: plugin)。

**理由**:
- Markdown 技能是提示词模板,无需沙箱验证,生成速度快
- Markdown 技能更安全,不涉及代码执行
- 大多数任务模式可以用提示词指导完成,无需编写代码
- Python 插件仅在确实需要复杂逻辑时才手动创建

**实现**:
- 在 `evolution.py` 的 `generate_skill_from_pattern()` 中明确要求生成 Markdown 格式
- 提示词中强调"不要生成 Python 代码"
- 沙箱验证仅在检测到 `type: plugin` 时才执行

**影响**:
- 自进化速度提升 ~80% (跳过沙箱验证)
- 生成的技能更易于用户理解和修改
- 降低了安全风险

---

## 2. 嵌入检索完全可选

**问题**: 嵌入模型是一个大依赖:
- 本地模型: ~80MB 下载,需要 sentence-transformers
- OpenAI API: 需要额外的 API 调用,增加成本

**决策**: 混合检索(TF-IDF + 嵌入)完全可选,默认使用简单的关键词匹配。

**理由**:
- 关键词匹配对大多数场景已经足够
- 保持"轻量级"的核心目标
- 用户可以根据需求选择是否启用混合检索
- 避免强制依赖大型库

**实现**:
```yaml
# config.yaml
memory:
  hybrid_retrieval:
    enabled: false  # 默认禁用
    provider: openai
    model: text-embedding-3-small
```

**权衡**:
| 方案 | 优点 | 缺点 |
|------|------|------|
| 关键词匹配 | 快速、零依赖、轻量 | 无法理解语义相似性 |
| 混合检索 | 更精确的语义理解 | 需要 API 或本地模型 |

**用户选择**:
- 默认: 关键词匹配 (零配置)
- 需要更高精度: 启用 OpenAI 嵌入 (已有 API key)
- 离线场景: 启用本地模型 (需安装 sentence-transformers)

**影响**:
- 核心依赖保持最小: openai + pyyaml
- 用户可以在不安装额外依赖的情况下使用所有核心功能
- 混合检索作为可选增强,按需启用

---

## 3. 为什么选择 Markdown 作为技能格式

**理由**:
- 人类可读,易于理解和修改
- 无需编译或解释,直接注入到 system prompt
- 版本控制友好(纯文本)
- 降低了技能创建的门槛

---

## 4. 为什么使用 SQLite 存储工作记忆

**理由**:
- 轻量级,无需额外的数据库服务
- Python 内置支持,零依赖
- 支持 SQL 查询,方便按时间范围检索
- 自动清理过期数据

---

## 5. 为什么情景/语义记忆使用 Markdown 文件

**理由**:
- 人类可读,用户可以直接编辑
- 支持 frontmatter 元数据
- 易于备份和迁移
- 无需数据库,降低复杂度

---

## 总结

LightHermes 的设计哲学是**轻量优先,按需增强**:
- 核心功能保持最小依赖
- 高级功能作为可选增强
- 优先使用简单方案,避免过度工程
- 保持代码量在 2000-5000 行范围内
