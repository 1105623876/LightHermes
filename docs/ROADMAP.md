# LightHermes 发展路线图

**版本**: v0.3.2 完成 + Phase 2 重排
**更新时间**: 2026-05-06
**设计理念**: 保持轻量，增强可用性，优先把分级记忆真正用好

---

## 设计原则

1. **轻量优先**: 核心代码保持在 3000 行以内
2. **渐进增强**: 高级功能可选，不增加基础依赖
3. **实用为主**: 优先解决真实使用场景的问题
4. **科研探索**: 在稳定基础上实验创新想法，但避免过早复杂化

## Phase 2: 记忆蒸馏与轻量架构收敛（v0.4.0）

**目标**: 删除过重的记忆图谱方向，围绕“分级记忆如何真正被用好”继续增强；同时参考 Hermes 的记忆生命周期与 nanobot 的轻量 agent/tool/skill/channel 架构，保持 LightHermes 简洁可维护。

**已完成起步项**:
- [x] 自进化成功质量评估：新增 `quality_score`、`quality_level`、`learning_worthy`、`quality_metrics`、`quality_version`
- [x] 成功模式分析只从高质量成功轨迹中学习，避免侥幸成功污染技能生成
- [x] 失败模式生成 `failure_report`，避免将失败轨迹固化为正向技能
- [x] 记忆系统四项增强：混合检索初始化修复、配置透传、工作记忆提升、压缩摘要入库

**参考项目取舍（2026-05-08 确认）**:
- LightAgent：借鉴工具注册、轻量 `MemoryProtocol` 和候选工具过滤思路；不照搬 `la_core.py` 的大单文件混合架构。
- Hermes：优先借鉴记忆生命周期、召回上下文 fencing、压缩摘要“仅作参考而非当前指令”的安全措辞；不照搬外部 memory provider 插件体系。
- nanobot：优先借鉴 `AgentHook`/`CompositeHook`、tool registry、schema 校验和 skill frontmatter 解析；暂缓 channel、cron、gateway 等生态能力。
- 当前推进顺序：先做记忆生命周期钩子，再做 `distill_memories()` 与容量治理，然后接入失败报告反模式索引，最后再拆 tool/skill 边界。

### 2.1 记忆蒸馏与容量治理
**优先级**: 高
**工作量**: 5-7 小时

**核心方向**:
- 工作记忆保存近期会话摘要和压缩结果，作为短周期上下文缓存
- 情景记忆保存项目/任务经验，强调“某次任务如何解决”
- 语义记忆只保存稳定事实、用户偏好、通用原则和反复验证的经验
- 删除“记忆图谱”计划，不引入图数据库或复杂实体关系层

**具体任务**:
- [x] 设计 `distill_memories()`：从工作/情景记忆中提炼高价值语义记忆
- [x] 增加记忆容量预算：对语义记忆设置字符/条目上限，避免无限追加
- [x] 增加重复与近重复检测：同类记忆优先合并而不是新增
- [x] 增加蒸馏元数据：`distilled_from`、`source_layer`、`confidence`、`last_verified`
- [x] 为归档、提升、蒸馏补边界测试，尤其是索引同步移除和幂等性

**参考 Hermes 的取舍**:
- 借鉴 frozen snapshot / bounded curated memory 思路，避免每次都把无限记忆塞进 prompt
- 借鉴 `on_pre_compress()` / `on_session_end()` 的时机，在压缩和会话结束时做提取
- 不照搬外部 memory provider 插件体系，先在现有四级记忆内闭环

### 2.2 分级记忆生命周期
**优先级**: 高
**工作量**: 4-5 小时

**目标**: 让短期/工作/情景/语义四层记忆有明确职责和迁移时机，而不是只做并列检索。

**生命周期钩子**:
- [x] `on_turn_start(query)`：准备召回上下文，并用 `<memory-context>` 安全包装，避免把召回内容当作新用户输入
- [x] `on_turn_end(user, assistant)`：同步助手回复到短期记忆，作为后续会话摘要来源
- [x] `on_pre_compress(messages)`：压缩前提取即将丢失的轻量线索，随压缩上下文进入摘要输入
- [x] `on_session_end(messages)`：CLI `/exit`、`/reset`、KeyboardInterrupt/EOF 时生成短期摘要并触发迁移/归档/提升
- [x] `on_memory_write(...)`：固定记忆或用户偏好写入时进入统一生命周期入口；当前先记录，后续扩展索引同步与审计

**实现原则**:
- 先做普通方法，不急着抽象 provider 插件
- 所有钩子失败都不阻断主对话
- 迁移和蒸馏必须幂等，重复运行不能产生重复记忆
- 保持默认关键词检索可用，语义检索仍作为可选增强

### 2.3 自进化与反模式学习
**优先级**: 中高
**工作量**: 4-6 小时

**已完成**:

- [x] 失败轨迹生成 `failure_report`，不再包装成正向技能

**下一步**:
- [x] 建立轻量反模式索引：让 `failure_report` 可以按任务类型/关键词召回

- [x] 将反模式学习结果优先存入情景记忆，稳定后再蒸馏到语义记忆

**暂缓**:
- 元学习机制保留为实验想法，不进入 v0.4.0 主线
- 反事实推理暂缓，避免为轻量框架引入过重决策记录系统

## 科研实验想法池

以下方向保留为想法池，不进入 v0.4.0 主线：

1. **元学习与迁移学习**
   - 从一个项目学到的技能迁移到另一个项目
   - 跨项目的知识共享机制
   - 个性化的智能体人格

2. **协作智能体**
   - 多个 LightHermes 实例协作完成任务
   - 任务分解与分配策略
   - 协作通信协议（轻量级）

3. **主动学习**
   - 智能体主动提问以获取更多信息
   - 不确定性估计与查询策略
   - 人机协作的最优策略

4. **可解释性**
   - 解释智能体的决策过程
   - 可视化记忆检索和技能选择
   - 生成决策报告

5. **安全与隐私**
   - 本地模型支持（Ollama 集成）
   - 敏感信息检测与过滤
   - 记忆加密存储

6. **代码理解增强**
   - AST 分析与代码图谱
   - 依赖关系可视化
   - 代码变更影响分析

## 实施策略

### 开发节奏
- **先收敛再扩展**: 每次只推进一个边界清晰的小功能
- **文档同步**: 计划、状态和测试说明随代码同步更新
- **参考不照搬**: Hermes / nanobot 作为架构参考，LightHermes 保持极简实现

### 质量保证
- **测试先行**: 关键记忆迁移、蒸馏、归档逻辑先补测试
- **边界验证**: 重点覆盖索引同步、幂等性、容量限制、失败回退
- **基线稳定**: 每个功能完成后跑 `pytest tests/`

## 成功指标

### 可用性指标
- CLI 启动时间 < 1s
- API 响应时间 < 2s（非流式）
- 记忆检索时间 < 100ms
- 测试覆盖率 > 80%

### 轻量性指标
- 核心代码尽量保持在 3000 行左右
- 核心依赖 < 5 个
- 安装包大小 < 10MB
- 内存占用 < 100MB

### Phase 2 指标
- 工作记忆、情景记忆、语义记忆迁移路径清晰且幂等
- 语义记忆具备容量治理和合并策略
- 失败报告可被检索并用于执行前风险提示
- agent/tool/skill/channel 边界比当前更清晰，但不引入重框架
